PVE-UPS — Benutzerhandbuch

Eine schlanke USV-Shutdown-Appliance für Proxmox VE: Sie überwacht eine oder mehrere netzwerkfähige USVs per SNMP und fährt bei Stromausfall die zugeordneten Proxmox-Hosts geordnet herunter — ganz ohne NUT und ohne von Hand gepflegte Konfigurationsdateien. Dieses Handbuch richtet sich an technisch versierte Administratoren, die das Produkt noch nicht kennen.

1 Was ist PVE-UPS?

PVE-UPS ersetzt klassische, herstellergebundene USV-Shutdown-Appliances (wie APC PowerChute Network Shutdown) für Proxmox-Umgebungen. Die Software läuft als ein einzelner, ressourcenschonender unprivileged LXC (~256 MB RAM) und benötigt nur ein Installer-Skript.

Kernfunktionen

  • Überwachung einer oder mehrerer USVs mit SNMP-Netzwerkkarte (Standard RFC 1628) per SNMP v1/v2c/v3 — reine Python-Implementierung.
  • Geordneter Shutdown der Hosts über ein Proxmox API-Token (Recht Sys.PowerMgmt) — kein Root-SSH nötig.
  • Host↔USV-Zuordnung mit Logik pro Host (UND/ODER) für redundante Netzteile bzw. aufgeteilte Last, inkl. Schaubild im Webinterface.
  • Komplette Einrichtung über einen Web-Wizard mit Test-Buttons.
  • Weboberfläche auf Englisch und Deutsch — automatisch passend zur Browsersprache.
  • Fail-safe: ein SNMP-Verbindungsverlust löst standardmäßig nur Alarm aus, keinen Shutdown.
  • Dry-Run-Modus für eine gefahrlose Inbetriebnahme.
  • Lesender REST/JSON-Status für Monitoring (/api/status, /api/health).

2 Voraussetzungen

  • Ein Proxmox VE-Host (Standalone — kein Cluster-/HA-Manager-Eingriff), auf dem der LXC laufen kann. Typischerweise einer der zu schützenden Hosts.
  • Eine oder mehrere USVs mit SNMP-Netzwerkkarte, die die Standard-RFC-1628-UPS-MIB per SNMP bereitstellen (herstellerunabhängig).
  • Netzwerkverbindung zwischen Appliance und USV (SNMP, Lesezugriff genügt) sowie zur Proxmox-API der zu schützenden Hosts (HTTPS, Port 8006).

3 Installation & Einrichtung

Die Installation erfolgt direkt aus der Proxmox-Node-Shell (Webinterface → Node → >_ Shell, als root). Das Skript lädt das aktuelle Release-Paket von GitHub nach, entpackt es und legt den LXC an:

bash -c "$(curl -fsSL https://github.com/ffind-dev/pve-ups/releases/latest/download/install.sh)"
# z.B. mit fester IP:
curl -fsSL https://github.com/ffind-dev/pve-ups/releases/latest/download/install.sh | bash -s -- \
  --ctid 950 --ip 10.0.0.50/24 --gateway 10.0.0.1 --hostname pve-usv

Anschließend das Webinterface unter http://<container-ip>:8080 öffnen und den Wizard durchlaufen:

  1. UI-Passwort setzen.
  2. USVs anlegen (SNMP-Adresse/Version) und je USV mit „SNMP testen" prüfen.
  3. Proxmox-Hosts eintragen (API-Token, siehe Abschnitt 4) und „Verbindung testen". Festlegen, welche USVs einen Host versorgen und nach welcher Logik (UND/ODER).
  4. Schwellwerte setzen bzw. die Standardwerte übernehmen.
  5. Optional den Benachrichtigungs-Webhook einrichten (HTTP-POST an ein Ticket-/Chat-/Monitoring-System).
Solange der Dry-Run aktiv ist, wird nichts heruntergefahren — ideal zum Testen. Erst wenn alles passt, den Dry-Run deaktivieren (Modus „SCHARF").
Läuft der LXC auf einem der geschützten Hosts, diesen in der Host-Liste als „Dieser Host" markieren — er wird dann garantiert zuletzt heruntergefahren.

4 Proxmox API-Token anlegen (pro Host)

Auf jedem Proxmox-Host einmalig einen dedizierten Benutzer mit ausschließlich dem Power-Recht und einen API-Token erzeugen:

pveum user add ups@pve
pveum role add UpsShutdown -privs "Sys.PowerMgmt"
pveum acl modify /nodes -user ups@pve -role UpsShutdown
pveum user token add ups@pve shutdown --privsep 0

Der letzte Befehl gibt Token-ID (ups@pve!shutdown) und Secret aus. Beides im Wizard unter „Proxmox-Hosts" eintragen und mit „Test" prüfen. Das Token ist jederzeit widerrufbar und ersetzt einen Root-SSH-Zugang.

5 SNMP an der USV

In der Weboberfläche der USV-Netzwerkkarte (meist unter Network → SNMPv1 bzw. SNMPv3) den Zugriff aktivieren und der Appliance-IP Lesezugriff geben. Es wird nur die Standard-RFC-1628-UPS-MIB gelesen (kein Schreibzugriff). Der Test im Wizard zeigt sofort Quelle (Netz/Akku), Akkustatus, Restlaufzeit und Ladestand an.

6 Funktion & Logik

Die Engine fragt alle USVs parallel ab und wertet jede USV unabhängig aus. Pro USV gibt es eine einfache Zustandsabfolge:

ONLINE Netzbetrieb  →  ON_BATTERY Stromausfall, USV im Akku  →  ausgelöst sobald eine Schwelle greift

Welcher Host fährt herunter?

Das entscheidet die Zuordnung jedes Hosts zu seinen USVs und seine Logik:

  • UND (ups_policy = all, Standard): Der Host fährt erst herunter, wenn alle ihm zugeordneten USVs ausgelöst haben — exakt das Verhalten für Server mit redundanten Netzteilen (je Netzteil eine USV). Solange eine USV noch Strom liefert, bleibt der Host an.
  • ODER (ups_policy = any): Der Host fährt herunter, sobald eine zugeordnete USV auslöst — für aufgeteilte, nicht redundante Last.

Mehrere fällige Hosts werden geordnet abgeschaltet. Der Host, der die Appliance selbst trägt (this_host), kommt zuletzt dran, damit er die übrigen noch abschalten kann. Tipp: den Appliance-Host an alle USVs hängen, damit er zuverlässig zuletzt fällig wird.

Erholt sich eine erforderliche USV (Netz kehrt zurück), wird ein noch nicht ausgeführter Shutdown wieder abgebrochen und der Zustand zurückgesetzt.

7 Fail-safe & Sonderfälle

Grundprinzip: Ein Verlust der SNMP-Verbindung ist kein bestätigter Stromausfall — die USV antwortet vielleicht nur kurz nicht, oder ein Switch dazwischen ist weg. Standardmäßig löst ein Kommunikationsverlust daher nur Alarm aus, keinen Shutdown. Zwei optionale Einstellungen verfeinern das:

  • Shutdown bei Verlust fortsetzen (Standard an): Bricht SNMP ab, nachdem bereits ein Akkubetrieb bestätigt war, läuft ein gestarteter Akkubetrieb-Countdown auf der lokalen Uhr weiter und löst bei Ablauf aus. Nur dieser zeitbasierte Auslöser überlebt den „blinden" Zustand (Restlaufzeit und Ladestand sind ohne SNMP unbekannt). Hatte ein Auslöser (z. B. Battery-Low oder Ladestand-Schwelle) dagegen schon vor dem Verbindungsabriss gegriffen, bleibt er bestehen und wird nicht auf den Zeitcountdown zurückgestuft. Countdown und ausgelöster Zustand überleben auch einen Neustart des Dienstes: beides wird auf Platte gemerkt und beim Start wiederhergestellt (max. 24 h alt).
  • Shutdown nach X Minuten reinem Verlust (Standard aus): Behandelt einen reinen Kommunikationsverlust (ohne vorher bestätigten Akkubetrieb) nach X Minuten doch als Ausfall. Bewusstes Opt-in für Umgebungen, in denen ein längerer SNMP-Ausfall als kritisch gilt.

Übersicht der Fälle

SituationVerhalten

8 Auslöser & Schwellen

Es genügt eine zutreffende Bedingung. Alle Werte sind im Wizard änderbar (Feld leeren = Bedingung aus); Schwellen lassen sich global setzen und pro USV überschreiben.

BedingungStandard
Akkubetrieb länger als600 s
Restlaufzeit unter10 min
Ladestand unter30 %
USV meldet battery low/depletedan

Poll-Intervall: 30 s im Netzbetrieb, 8 s im Akkubetrieb.

Battery-Low, Restlaufzeit- und Ladestand-Schwelle lösen sofort aus — unabhängig vom Zeit-Countdown. Der im Dashboard angezeigte Countdown betrifft ausschließlich die Bedingung „Akkubetrieb länger als" und verschwindet, sobald die USV ausgelöst hat. Ob ein Host daraufhin sofort herunterfährt, entscheidet seine UND-/ODER-Logik (siehe Funktion & Logik): bei UND wartet er, bis alle ihn versorgenden USVs ausgelöst haben.

9 Monitoring / REST

Zwei lesende Endpunkte stehen ohne Authentifizierung und ohne Secrets zur Verfügung:

  • GET /api/status — kompletter Zustand: Appliance, alle USVs, Hosts, Shutdown-Status sowie die Ereignisse der letzten 48 h inkl. Severity-Zusammenfassung.
  • GET /api/health — 200, wenn die Engine läuft (mit Version und Erreichbarkeit der USVs), sonst 503. Ideal für Uptime-Checks.
curl http://<container-ip>:8080/api/status | jq
# nur kritische Ereignisse der letzten 48 h:
curl -s http://<container-ip>:8080/api/status \
  | jq '.events[] | select(.severity=="critical")'

Alle ändernden Endpunkte (Wizard speichern, Test-Shutdown) sind passwortgeschützt.

Ereignisprotokoll, Webhook-Nachrichten und API-Meldungen sind einheitlich englisch — unabhängig von der gewählten Oberflächensprache. Das hält Auswertung und Monitoring über Installationen hinweg konsistent.

10 Updates & Betrieb

Updates lassen sich direkt im Webinterface hochladen; ein schmaler, privilegierter Begleitdienst wendet sie an (die App selbst bleibt unprivilegiert). Nach dem Upload wird das Update typischerweise innerhalb weniger Sekunden eingespielt. Über das Webinterface können zudem Konfiguration exportiert/importiert, NTP-Server und Zeitzone gesetzt und das Ereignisprotokoll eingesehen/geleert werden.

Hinweis: Die Zeitzone (Abschnitt System, z.B. Europe/Berlin) sollte gesetzt werden, da der Container sonst in UTC läuft. Die Selbsttest-Uhrzeit des täglichen Proxmox-Verbindungstests gilt in dieser lokalen Zeit; nach dem Setzen der Zeitzone startet der Dienst kurz neu.

Update von Version 2.x auf 3.x

Das jeweils aktuelle Release-Paket (pve-usv-<version>.tar.gz) von der GitHub-Release-Seite herunterladen und im Webinterface unter „Update" hochladen — die Konfiguration bleibt vollständig erhalten. Zwei Verhaltensänderungen: die E-Mail-Benachrichtigung entfällt (ein evtl. konfigurierter SMTP-Eintrag wird beim nächsten Speichern automatisch entfernt; der Webhook bleibt), und Ereignis-/ Webhook-Texte sind ab 3.0.0 englisch.

11 Fehlerbehebung

Update bleibt in der Warteschlange

Bleibt ein hochgeladenes Update bei „⏳ In Warteschlange" stehen, ist der Queue-Drainer auf dieser Instanz nicht aktiv. Das passiert einmalig bei Boxen, die aus einer sehr alten Version aktualisiert wurden. Einmalig im Container ausführen (aktiviert den Dienst dauerhaft):

pct exec <CTID> -- systemctl start pve-usv-agent.service
# oder direkt im Container:
systemctl start pve-usv-agent.service

USV nicht erreichbar / Alarm

SNMP-Adresse, Version/Community bzw. v3-Credentials und die Firewall zwischen Appliance und USV prüfen. Im Wizard „SNMP testen" zeigt die genaue Ursache. Ein Kommunikationsverlust allein fährt nichts herunter (Fail-safe).

Interne Namen: pve-usv

Der Produktname ist PVE-UPS, technisch heißen Dienst und Pfade aber weiterhin pve-usv — also z. B. systemctl status pve-usv, /etc/pve-usv/config.yaml und /var/lib/pve-usv/. Das ist beabsichtigt und hält Updates bestehender Installationen kompatibel.
